● Drawflow 的问题：

  1. DOM 污染 — Drawflow 把画布节点直接插到 DOM 里并用自己的 CSS，会和现有 UI
  冲突。我们这个项目有 835 行的 style.css + Flask/Jinja 模板，Drawflow
  的样式隔离很差。
  2. 没 state 管理 — 节点/连线数据存在 DOM 属性里，不是 JS
  对象。保存/加载画布时你得自己遍历 DOM 提取 state，很容易漏掉。
  3. 扩展性天花板 — 要自定义节点渲染就得直接改 DOM，没法用组件化思维。以后加 AI
  类型检查的端口标注、报错高亮，都会很痛苦。
  4. 社区停滞 — 维护不活跃，PR 堆积无人合并。

  React Flow 的优势对应：

  1. 数据驱动 — 节点和边是纯 JS 对象，存/加载就是 JSON 序列化，天然适配你的画布
  → JSON → LLM 代码生成这条链路。
  2. 样式隔离 — React 组件 + CSS-in-JS，不影响现有 UI。
  3. 自定义节点是 React 组件 —
  报错边框、端口类型颜色、连线校验失败提示，都是组件级别的修改。
  4. 社区活跃 — 长期维护有保障。

  但这个项目目前是 Flask + 原生 JS，引入 React Flow 需要决策：

  - 要么整个前端迁移到 React（大工程，P0 不推荐）
  - 要么把画布区域做成独立小 React 应用，通过 window 事件和现有 JS 通信（P0
  可行方案）

  我的推荐： P0 选 Drawflow，但接受它的限制。因为你的 P0
  只是"搭起来"——拖拽、连线、存 JSON、读 JSON——Drawflow 够用，集成成本低。等 P1
  加 AI 代码生成、P2 加类型检查时，再评估是否值得迁移到 React Flow。

  同意 Drawflow 做 P0 的选型吗？
