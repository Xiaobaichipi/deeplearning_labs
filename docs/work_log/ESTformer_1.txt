❯ ## ETSformer 集成方案审查                                                     
                                                                                
  整体方向正确，但在几个关键细节上与已确立的模式存在偏差。逐一分析。            
                                                                                
  ---                                                                           
                                                                                
  ### ✅ 正确的决策                                                             
                                                                                
  | 决策 | 评价 |                                                               
  |------|------|                                                               
  | Pipeline = large | ETSformer 需要 x_mark、dec_inp，4 参数 forward           
  签名，正确 |                                                                  
  | 包结构 = 自包含 `etsformer_layers/` | 与 Autoformer、Crossformer 一致 |     
  | `d_layers` 隐藏、自动等于 `e_layers` | Crossformer 已采用相同策略（Decoder  
  层数 = Encoder 层数 + 1），合理 |                                             
  | `top_k` 作为独立参数名 | ETSformer 的 top-k 是频域分量选择，语义上不同于    
  Autoformer 的 attention factor，保留独特命名是对的 |                          
  | 暴露 7 个参数 | 与已有多数大模型对齐 |                                      
  | `Transform` 增强包含 | 这是 ETSformer 原始论文的组成部分，不加入反而缺功能  
  |                                                                             
                                                                                
  ---                                                                           
                                                                                
  ### ⚠️ 需要修正的问题                                                         
                                                                                
  #### 问题 1：Embedding 来源打破了自包含原则                                   
                                                                                
  方案中标注 **"Embedding 来源: shared_layers.Embed.DataEmbedding"**，但这与    
  Crossformer 集成时确立的模式冲突。                                            
                                                                                
  回顾当时的决策：                                                              
                                                                                
  > Crossformer: 方案 A（自包含包）— "无耦合风险、方便从上游同步更新、与        
  Autoformer 模式一致"                                                          
                                                                                
  ETSformer 如果依赖                                                            
  `shared_layers/Embed.py`，那就不是真正的自包含包——`etsformer_layers/`         
  有了外部依赖，未来从 `time_series_models_labs`                                
  同步更新时需要额外注意这个外部引用。                                          
                                                                                
  **修正**：ETSformer 应拥有自己的 `etsformer_layers/Embed.py`，包含它需要的    
  `DataEmbedding`。这与 Crossformer 的                                          
  `crossformer_layers/Embed.py`（独立复制了 `PatchEmbedding`）模式一致。不依赖  
  `shared_layers/`。                                                            
                                                                                
  **理由**：                                                                    
  - 保持所有大模型层包的独立性                                                  
  - 未来从上游同步时只需整体替换目录                                            
  - 只有 Informer 使用 `shared_layers/`（历史原因），不应扩大这个例外           
                                                                                
  ---                                                                           
                                                                                
  #### 问题 2：`activation` 新增 `sigmoid` 选项，前端需适配                     
                                                                                
  之前所有模型（Autoformer/Informer/Crossformer/DLinear）的激活函数只有 `gelu`  
  / `relu`。ETSformer 首次引入 `sigmoid`。                                      
                                                                                
  这影响：                                                                      
                                                                                
  1. **HTML 模板**：ETSformer 的 `activation` 下拉框需要三个选项：              
     ```html                                                                    
     <option value="sigmoid" selected>Sigmoid</option>                          
     <option value="gelu">GELU</option>                                         
     <option value="relu">ReLU</option>                                         
     ```                                                                        
     `sigmoid` 应为默认选中（ETSformer 原始默认）。                             
                                                                                
  2. **JS 测试**：`vitest.setup.js` 中需要添加 `etsformerParams` 面板的 DOM     
  片段，`ui.test.js` 和 `app.test.js` 需要增加对应的测试用例。                  
                                                                                
  3. **后端**：`CrossformerWrapper.__init__` 中 `activation` 参数已支持         
  `gelu`/`relu` 切换，ETSformerWrapper 需要额外处理 `sigmoid`                   
  的情况（`nn.Sigmoid()`）。                                                    
                                                                                
  **不是问题，但需要在变更清单中明确标注前端和后端的改动点。**                  
                                                                                
  ---                                                                           
                                                                                
  #### 问题 3：`scipy` 依赖需确认

  `requirements.txt` 中是否已有 `scipy`。                                       
                                                                                
  - 如果**没有**，需要新增 `scipy>=1.7.0`（`next_fast_len` 在 scipy 1.0+        
  已存在，1.7+ 更稳定）。                                                       
  - 如果**已有**，确认版本是否足够。                                            
                                                                                
  **建议**：在实施前先 `pip list | grep scipy` 确认。如果缺失，更新             
  `requirements.txt` 并在方案中标注。
